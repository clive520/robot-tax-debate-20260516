# AI 辯論網站運作流程

本文件用來定義「AI 辯論入口網站」之後新增辯論議題時的標準流程。目標是讓每一場辯論都能用一致的規格完成：由使用者提供題目，AI 模型分配角色，產生完整辯論與裁判評分，整理成 Markdown，製作成可閱讀與可播放語音的網頁，最後部署到 GitHub Pages。

## 一、核心模式

使用者提供一個辯論題目後，系統從 AI 角色池中分配三個角色：

| 角色 | 任務 |
| --- | --- |
| 正方 | 支持命題，提出定義、核心理由、反駁與結辯 |
| 反方 | 反對命題，提出定義、核心理由、反駁與結辯 |
| 裁判 | 閱讀雙方完整辯論，依多個面向評分，判定勝負並寫出理由 |

目前預設候選池：

| AI | 建議定位 |
| --- | --- |
| Codex / OpenAI | 結構化論證、政策分析、網頁製作與部署流程 |
| Gemini | 多模態與大型脈絡理解，適合擔任正反方辯手 |
| Claude | 長文理解、語氣細膩與倫理判斷，適合擔任裁判或辯手 |
| xAI Grok | 適合擔任正反方，特別是提出尖銳反例、質疑假設與加強攻防張力 |
| DeepSeek | 適合擔任正反方或推理型裁判，能協助產生多輪推理與制度比較 |
| Microsoft Copilot | 適合擔任裁判或實務型辯手，特別是教育、企業、行政與政策可行性議題 |

## 二、角色分配規則

每次新增辯論時，原則上採隨機分配：

1. 從可用 AI 角色池中抽出三個模型。
2. 隨機指定為正方、反方、裁判。
3. 同一場辯論中，裁判不可同時擔任正方或反方。
4. 若某個模型當下無法登入、無法使用或回應品質不穩，改由備選模型補位。
5. 若使用者指定某個模型擔任特定角色，則以使用者指定為優先。
6. 若議題屬於高度專業或高風險領域，裁判應優先選擇擅長長文審閱、價值衝突分析與公平評分的模型。

建議保留一份 `role-history.md` 或類似紀錄，記下每一場辯論的角色分配，避免長期下來同一模型過度固定在某一方。

## 三、辯論輪次

每場辯論固定分成以下階段：

1. 正方申論：提出正方核心觀點與主要理由，包含破題、定義、2 到 3 個重點。
2. 反方申論：提出反方核心觀點與主要理由，包含破題、定義、2 到 3 個重點。
3. 正方駁論：針對反方論點找出漏洞進行反駁，並補充己方證據。
4. 反方駁論：針對正方論點進行反駁，並補充己方證據。
5. 反方結辯：反方先總結己方論點，指出對方矛盾，做最後拉票。
6. 正方結辯：正方後總結己方論點，指出對方矛盾，做最後拉票。
7. 裁判評分：裁判依評分表給分、寫出勝負理由與改進建議。

## 四、裁判評分規格

裁判評分應採多面向評分，不只給出總分。建議評分表如下：

| 面向 | 說明 | 建議分數 |
| --- | --- | --- |
| 立論清晰度 | 是否清楚定義命題、界定討論範圍、提出核心主張 | 10 |
| 論證深度 | 理由是否完整，是否能處理制度、倫理、實務與反例 | 15 |
| 證據與例子 | 是否提供具體案例、合理資料或可驗證的推論 | 10 |
| 反駁有效性 | 是否真正回應對方核心論點，而不是只重複己方立場 | 15 |
| 政策可行性 | 若議題涉及制度設計，是否提出可執行方案與配套 | 10 |
| 結辯說服力 | 是否能收束全場、凸顯勝負關鍵 | 10 |
| 整體策略 | 論點安排、攻防節奏、矛盾處理是否成熟 | 10 |

總分可依議題需要調整為 70、80 或 100 分制，但同一場辯論中正反雙方必須使用同一套標準。

裁判必須輸出：

1. 正方各面向分數。
2. 反方各面向分數。
3. 總分與勝方。
4. 勝負理由。
5. 雙方最強論點。
6. 雙方最大弱點。
7. 若要重賽，雙方應如何補強。

## 五、內容產出流程

每次新增辯論時，依序完成：

1. 確認題目與角色分配。
2. 請正方產出正方申論。
3. 請反方閱讀正方申論後產出反方申論。
4. 請正方閱讀反方申論後產出正方駁論。
5. 請反方閱讀正方駁論後產出反方駁論。
6. 請反方產出反方結辯。
7. 請正方閱讀反方結辯後產出正方結辯。
8. 將完整辯論交給裁判評分。
9. 整理成 `debate.md`，保留每一輪的角色、標題、發言內容與裁判評語。
10. 製作辯論網頁，清楚區分正方、反方與裁判。
11. 為正方、反方、裁判分別產生不同聲音的語音檔。
12. 製作 Podcast 版音訊，只保留正方與反方辯論，不包含裁判評分。
13. 上架或排程 YouTube 與 Podcast，取得可公開的影片資訊，並記錄 Spotify 節目頁。
14. 在該篇辯論網頁內嵌 YouTube 播放器；Podcast 區保留本機音訊播放，並固定連到 Spotify「AI 辯論所」節目頁，不嵌入 Spotify 單集播放器。
15. 更新首頁入口卡片或發布資料。
16. 本機檢查畫面、網頁音訊、Podcast 音訊與影音內嵌區。
17. Commit 並 push 到 GitHub。
18. 確認 GitHub Pages 網址可正常開啟。

## 六、檔案與資料夾規格

每一場辯論建議建立一個獨立資料夾：

```text
debates/
  topic-slug/
    index.html
    debate.md
    app.js
    generate-audio.py
    audio/
      positive-*.mp3
      negative-*.mp3
      judge-*.mp3
    podcast/
      debate-podcast.mp3
      podcast-notes.md
```

首頁維護於：

```text
index.html
styles.css
```

文件與流程規格維護於：

```text
docs/
  debate-workflow.md
  publishing-schedule.md
  work-log.md
```

## 七、工作日誌規則

專案必須維護 `docs/work-log.md` 作為正式工作日誌。之後每一項功能、內容、資料庫、登入、部署或設計調整，都必須在工作日誌中新增紀錄。

每筆工作日誌至少包含：

1. 日期。
2. 變更類型，例如 `content`、`frontend`、`backend`、`docs`、`deploy`、`ops`。
3. 摘要。
4. 影響範圍。
5. 驗證狀態。
6. 後續事項。

若變更涉及 Supabase、Google Cloud、GitHub Pages 或其他外部服務，只記錄設定目的與結果，不記錄密碼、OAuth secret、service role key 或其他不可公開的憑證。

命名建議：

1. `topic-slug` 使用英文小寫與連字號，例如 `robot-tax`、`euthanasia`、`school-phone`。
2. Markdown 中保留中文原題。
3. 首頁卡片顯示中文題目、簡短摘要、角色分配、裁判結果與進入按鈕。

## 八、語音規格

每一場辯論至少要有三種不同聲音：

| 角色 | 聲音策略 |
| --- | --- |
| 正方 | 穩定、清楚、有說服力 |
| 反方 | 與正方明顯不同，可稍微更冷靜或尖銳 |
| 裁判 | 中立、節奏較穩、像正式評審 |

目前可延續的語音配置：

| 角色 | Edge TTS 聲音 |
| --- | --- |
| 正方 | `zh-TW-YunJheNeural` |
| 反方 | `zh-TW-HsiaoChenNeural` |
| 裁判 | `zh-TW-HsiaoYuNeural` |

語音製作時應注意：

1. 不同角色使用不同聲線。
2. 段落不要過長，避免聽起來像機械朗讀。
3. 可在產生語音前微調標點，增加自然停頓。
4. 網頁上應能逐段播放，也能讓讀者清楚知道目前播放的是哪一方。

## 九、Podcast 產出規格

每一篇辯論完成後，除了網站版本，也要規劃一份 Podcast 版本，供之後上傳到 YouTube 的 Podcast 頻道。

Podcast 版本與網站版本不同：

| 項目 | 網站版本 | Podcast 版本 |
| --- | --- | --- |
| 正方內容 | 包含 | 包含 |
| 反方內容 | 包含 | 包含 |
| 裁判評分 | 包含 | 不包含 |
| 留言與互動 | 包含 | 不包含 |
| 主要用途 | 閱讀、查詢、聽單段語音 | 連續收聽完整正反辯論 |

Podcast 音訊應包含：

1. 簡短開場，說明辯論題目。
2. 正方申論。
3. 反方申論。
4. 正方駁論。
5. 反方駁論。
6. 反方結辯。
7. 正方結辯。
8. 簡短結尾，引導聽眾到網站閱讀裁判評分與留言。

Podcast 音訊不得包含：

1. 裁判評分內容。
2. 裁判勝負判定。
3. 網站留言、按讚或管理資訊。

每場辯論的 Podcast 檔案建議放在：

```text
debates/
  topic-slug/
    podcast/
      debate-podcast.mp3
      podcast-notes.md
```

`podcast-notes.md` 建議記錄：

1. YouTube 標題。
2. YouTube 說明文字。
3. 辯論題目。
4. 正方與反方角色。
5. 網站文章連結。
6. 是否已上傳 YouTube。
7. YouTube 影片或 Podcast 網址。

初期可先手動上傳 YouTube。未來若流程穩定，再評估是否自動產生 YouTube 上傳素材、封面圖、章節時間戳與描述文字。

### Podcast 影片製作準則

每一篇辯論若要製作 YouTube 影片版，應沿用下列流程，避免音訊、字幕與畫面各自產生而導致時間軸不一致。

#### 產出順序

1. 先完成網站版辯論文字與正反方六個段落內容。
2. 使用 `scripts/build-synced-podcast.py <slug>` 產生 Podcast 音訊、`captions-source.json` 與 YouTube 用 `captions.srt`。
3. 使用 `scripts/create-podcast-video.py <slug>` 產生 `podcast-video.mp4`。
4. 從影片擷取至少三張預覽圖，建議包含正方、反方與結辯段落。
5. 檢查音訊、SRT 結尾時間、影片長度與預覽圖版面。
6. 上傳 YouTube 時，使用 `podcast-video.mp4` 搭配 `captions.srt`，不要把逐字字幕直接燒進影片。

#### 字幕與音訊規則

1. Podcast 只包含正方與反方辯論，不包含裁判評分與勝負判定。
2. 字幕以文句段落為主，不採固定 15 字硬切；若句子過長，再依逗號、頓號、分號、冒號等自然語氣點拆分。
3. 時間軸必須由每段 TTS 音訊實際長度累積產生，避免用全文字數比例推估。
4. 正方、反方應使用不同聲音，讓聽眾能分辨目前立場。
5. 若分段 TTS 尾端有固定停頓，可用固定尾端裁切參數處理；目前預設為每段最多裁切約 `0.45` 秒。

#### 影片畫面規則

1. 影片比例固定為 16:9，解析度 1280 x 720。
2. 影片不燒逐字字幕，底部保留乾淨安全區，供 YouTube 載入 SRT 字幕。
3. 畫面需顯示辯論題目、進度列、段落重點、視覺提示與 AI 主持人。
4. 人像下方需顯示發言狀態牌，包含「目前發言」、AI 名稱與辯論段落，例如 `Claude / 正方申論`。
5. 發言狀態牌上緣應與左側重點文字框上緣對齊，避免漂浮感，也讓觀眾快速辨識目前段落。
6. 右側人像與狀態牌不得壓到底部字幕安全區；左側重點文字不得與右側狀態牌重疊。
7. 若未來加入照片或影片素材，素材應服務於當前段落重點，不應干擾字幕安全區或角色標示。

#### 檔案位置

```text
debates/
  topic-slug/
    podcast/
      debate-podcast.mp3
      captions-source.json
      captions.srt
    video/
      output/
        podcast-video.mp4
        captions.srt
        preview-speaker-30s.jpg
        preview-speaker-190s.jpg
        preview-speaker-470s.jpg
```

`debates/<slug>/video/output/` 與 `debates/<slug>/podcast/caption-segments/` 屬於本機產出資料夾，不納入 Git。流程腳本、辯論文字、網站頁面與工作日誌才需要提交。

#### 驗證標準

1. `debate-podcast.mp3` 可完整播放，且涵蓋六個正反方段落。
2. `captions.srt` 最後一格時間應接近 Podcast 音訊總長度。
3. `podcast-video.mp4` 長度應接近 Podcast 音訊長度。
4. 預覽圖需至少檢查正方、反方、結辯三種狀態。
5. 預覽圖需確認底部字幕安全區乾淨。
6. 發言狀態牌需正確顯示目前 AI 名稱與辯論段落。
7. 影片與字幕上傳 YouTube 前，需抽查中段字幕是否仍與聲音相符。

## 十、延伸 AI 角色池

目前正式候選池已包含 Codex / OpenAI、Gemini、Claude、xAI Grok、DeepSeek、Microsoft Copilot。以下補充各模型適合的角色定位，並列出未來可再納入的延伸候選。實際使用前應確認帳號可用性、使用限制與輸出品質。

| AI / 服務 | 適合角色 | 適合原因 | 注意事項 |
| --- | --- | --- | --- |
| Perplexity Sonar | 資料查核員、證據裁判、輔助評審 | 內建網頁搜尋與引用，適合查證時事、政策資料與來源可信度 | 較適合作為「資料裁判」或輔助評審，不一定要擔任主要辯手 |
| Mistral Le Chat | 正方、反方、備用裁判 | 支援多語言，適合提供不同模型家族的觀點，能做摘要、研究與文件審閱 | 若議題需要台灣脈絡，提示詞要明確要求使用繁體中文與在地情境 |
| xAI Grok | 正方、反方、交叉質詢、反例產生者 | 適合產生尖銳反問、替對方找漏洞，增加攻防張力 | 即時資訊需啟用搜尋工具；未啟用時不應依賴它處理最新事實 |
| DeepSeek | 正方、反方、推理型裁判 | 適合成本敏感、需要多輪推理或大量草稿比較的場景 | 涉及敏感政治、法規或在地資料時，應另外做來源查核 |
| Microsoft Copilot | 正方、反方、實務型裁判、教育或企業情境評論 | 與 Microsoft 365、生產力工具和網頁資料整合度高，適合評估實務可行性 | 若使用組織帳號，需留意資料權限、授權與隱私設定 |
| Meta Llama 系列 | 開源 / 本機部署辯手 | 適合建立可控、可重現、低依賴平台帳號的備用辯手 | 需要自行選擇託管方式與模型版本，品質會受部署環境影響 |

建議的角色池策略：

1. 基本候選池：Codex / OpenAI、Gemini、Claude、xAI Grok、DeepSeek、Microsoft Copilot。
2. 資料查核局：加入 Perplexity 作為資料查核員，但裁判仍可由任一正式候選模型擔任。
3. 高攻防局：優先讓 Grok 或 DeepSeek 擔任其中一方，負責提出更強反例。
4. 實務評估局：可讓 Copilot 擔任裁判，強化教育、企業、行政與政策可行性的判斷。
5. 高可信度局：由 Perplexity 先列來源，再讓裁判依來源與辯論表現評分。
6. 本機備援局：若網頁模型無法使用，可用 Llama 類模型作為臨時辯手，但裁判結果應標註模型來源。

## 十一、品質控管

每場辯論完成前，至少檢查：

1. 命題是否被正反雙方用同一個定義討論。
2. 雙方是否都有申論、駁論、結辯。
3. 裁判是否真的比較雙方，而不是只摘要內容。
4. 分數是否加總正確。
5. 網頁是否清楚標示正方、反方、裁判。
6. 音訊是否能播放，且三方聲音可辨識。
7. Podcast 版是否只包含正反方，不包含裁判內容。
8. Podcast 音訊是否可正常播放。
9. 首頁入口是否連到正確頁面。
10. GitHub Pages 網址是否部署成功。

若議題涉及法律、醫療、政治、重大公共政策或最新事件，應補上資料來源，並在裁判或註解中說明「哪些內容是辯論推論，哪些內容有外部來源支持」。

## 十二、建議 Prompt 模板

### 角色分配

```text
請從以下 AI 角色池中隨機分配正方、反方、裁判：
Codex / OpenAI、Gemini、Claude、xAI Grok、DeepSeek、Microsoft Copilot。

規則：
1. 正方、反方、裁判不可重複。
2. 若使用者指定角色，以使用者指定為準。
3. 若正式候選池中有模型無法使用，可從 Perplexity、Mistral Le Chat、Meta Llama 系列補位。
4. 請輸出角色分配與理由。
```

### 辯手

```text
你現在是本場辯論的【正方 / 反方】。
題目是：「{辯論題目}」
請依照目前輪次完成發言。
請使用繁體中文，語氣正式、有說服力。
若是駁論或結辯，必須具體回應對方剛才的論點。
```

### 裁判

```text
你現在是本場辯論的裁判。
請公平閱讀正方與反方所有發言。
請依照以下面向評分：
立論清晰度、論證深度、證據與例子、反駁有效性、政策可行性、結辯說服力、整體策略。
請輸出分數表、總分、勝方、勝負理由、雙方優缺點與改進建議。
```
